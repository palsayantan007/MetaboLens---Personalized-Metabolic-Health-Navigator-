# Implementation Plan: MetaboLens - Personalized Metabolic Health Navigator

This implementation plan is based on the specifications detailed in `design.md`, `requirements.md`, and `tasks.md`. It outlines a concrete strategy to build the MetaboLens prototype, focusing on a robust data strategy, ML model integration, and a production-ready web architecture.


## 1. Data Strategy & Research Context

As this must be a research-backed prototype, acquiring or simulating high-quality data is critical. The requirements specify:
- **Input:** 103 cytokines + 14 clinical markers
- **Study References:** 1,982 samples from 64 subjects, Th1/Th17 immune markers.

1. **Pre-computed Embeddings & Clusters (Real Data):** We will use the existing processed outputs from the provided research dataset. The `data/` directory contains:
   - `embeddings_original_256d.csv`: The initial 256-dimensional embeddings generated by the multi-modal contrastive model.
   - `embeddings.csv`: The final UMAP-reduced 64-dimensional embeddings across the ~1,982 samples.
   - `cluster_summary.csv`: The 28 distinct health state clusters (with noise mappings) discovered by HDBSCAN, exactly mirroring the `0.747 baseline_silhouette` target described in the research paper's performance metrics.
2. **Public Data Sources (Grounding & Validation):** To ensure the project aligns with large publicly available datasets, we will utilize and reference the following sources for our clinical ranges and cytokine profiles:
   - **Integrated Human Microbiome Project (iHMP):** Specifically the prediabetes study cohort, which provides extensive longitudinal multi-omics data, including 100+ circulating cytokines/chemokines, microbiota, and clinical markers (fasting glucose, HbA1c, SSPG).
   - **NHANES (National Health and Nutrition Examination Survey):** To establish realistic distributions and correlations for standard clinical metabolic markers across a diverse population.
3. **Personas Construction:** Rather than creating synthetic personas, we will sample directly from the 28 discovered clusters, which are representative of the public data profiles (e.g., normal glucose tolerance to diabetes). We will map representative baseline (e.g., Cluster 0), intermediate (e.g., Clusters 9, 14), and severe diabetic states (e.g., Cluster 27) directly to specific embeddings in the dataset.

## 2. Machine Learning Architecture

The core of MetaboLens is a multi-modal contrastive learning model that outputs a 128-dimensional embedding.

1. **Contrastive Learning Architecture:** The underlying model processing the data utilizes a multi-modal PyTorch neural network combining Cytokine (Transformer), Clinical (Feed-Forward), and Temporal (Feed-Forward) branches. For this implementation phase, **we will utilize the pre-calculated outputs of this architecture** (`embeddings.csv`) to serve the application instantly, focusing our API on mapping the user input/selected persona directly to the pre-computed HDBSCAN clusters and SHAP explainability matrices.
2. **Clustering, Interpretability & Visualization:**
   - **UMAP & HDBSCAN Integration:** The backend will expose the existing 28 clusters discovered by HDBSCAN.
   - **SHAP (SHapley Additive exPlanations):** We will map the top meaningful impactful biomarkers (risk factors) to the representative personas.
   - **Interactive 2D Visualization:** Further reducing the 64D/256D embeddings using UMAP purely for the 2D "Health Map" frontend scatter plot.

## 3. Technology Stack & Component Design

### Backend (Python / FastAPI)
- **FastAPI:** High-performance, async RESTful API.
- **PyTorch & scikit-learn:** For model inference, normalization, and HDBSCAN clustering.
- **SHAP & UMAP-learn:** For explainability and 2D projections.
- **ReportLab:** To generate the PDF Clinician Reports.
- **Architecture:** Following `design.md`, we will structure the backend into: `EmbeddingService`, `AnalysisService`, `VisualizationService`, and `ReportService`.

### Frontend (React / TypeScript / Vite)
- **Vite + React (TypeScript):** For a fast, strictly-typed single-page application.
- **Tailwind CSS:** To build a premium, modern, and highly responsive UI (glassmorphism, clean cards, health state color-coding: Green/Yellow/Red).
- **D3.js:** To build the highly interactive "Health Map" (scatter plot with zoom, pan, and cluster regions).
- **Recharts:** For the Temporal Progression timeline, showing risk scores over time.

## 4. Execution Phases

We will build the system in the following phases, mapped to the task breakdown in `tasks.md`.

### Phase 1: Foundation & Data Integration (Tasks 1-2, 11)
- Initialize the frontend and backend project structures.
- Implement data ingress pipelines to load and serve `embeddings.csv`, `embeddings_original_256d.csv`, and `cluster_summary.csv`.
- Create the **Persona Service** that strategically selects 5 highly representative patients from the real 1,982 samples to define the hackathon stories (Healthy, Pre-diabetic, Immune Dysregulation, Severe).

### Phase 2: Backend API & Explanation Layer (Tasks 3-6, 12)
- Re-map the existing `EmbeddingService` from generation requirements to *retrieval and validation* requirements from the CSV sources.
- Integrate SHAP mockups or compute SHAP values for the specific persona slices.
- Expose these services through the FastAPI endpoints (`/api/analyze`, `/api/health-map`, `/api/personas`).

### Phase 3: Frontend Dashboard & Visualizations (Tasks 14-17)
- **Data Ingestion Portal (For Clinicians):** Build a clear, intuitive interface where a doctor can upload a patient's multi-omics CSV or manually input key clinical/cytokine values. The UI must clearly validate `103 cytokines + 14 clinical markers`.
- **Clinician-Friendly Dashboard:** Build the main React Dashboard with an emphasis on actionable, plain-English medical summaries designed for doctors. Avoid overwhelming the user with raw embeddings; focus heavily on the Risk Score and explicit Recommendations.
- **Interpretability Visualizations:**
  - Implement the `HealthMapVisualizer` using D3.js to render the 2D UMAP coordinates and temporal trajectories.
  - Build the `RiskFactorPanel` and `RecommendationsPanel` to interpret the SHAP values (color-coding elevated/reduced biomarkers).
  - Implement the `TemporalProgressionChart` using Recharts to track patient history.

### Phase 4: Reporting & Polish (Tasks 18-22, 25)
- Implement PDF report generation using ReportLab in the backend.
- Build the `ModelInfoPanel` detailing the model's metrics and data sources.
- Add loading states, error handling boundaries, and export features.
- Conduct End-to-End integration testing.

## Verification Plan

1. **Automated Unit & Property Tests:**
   Using Python's `hypothesis` library (as outlined in the specs), we will run property-based tests verifying structural correctness (e.g., embeddings are exactly 128D, risk scores map properly to [0,1], missing data does not crash inference).
2. **Visual & Interactive Verification:**
   We will manually test the D3.js Health Map rendering, ensuring that clusters separate visually and the 5 personas map to distinct regions.
3. **End-to-End API Testing:**
   We will use standard HTTP clients to test the REST endpoints, ensuring latency constraints (< 1 second inference, < 3 seconds end-to-end) are met.
